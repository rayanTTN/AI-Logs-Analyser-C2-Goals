#1 Explain the IP request surge mechanism (Also add here which module is required in the lambda layer for this mechanism to work)
The Machine Learning Approach: Isolation Forest
Since we do not have historical data to "train" a model on what a DDoS attack looks like, we must use Unsupervised Learning. The industry standard for this is the Isolation Forest algorithm from the scikit-learn library.
How it will work in your Lambda:
Time-Bucketing (Feature Engineering): Instead of just counting total hits, we will group the logs into 1-minute intervals. We will feed the model an array of data like: [IP, Minute, Request_Count].
The ML Evaluation: The Isolation Forest will analyze the entire batch. It builds "trees" to isolate data points. If an IP suddenly spikes from 10 requests to 300 requests in a single minute, the model will instantly isolate it as a statistical outlier.
The Output: The model assigns a score of 1 (Normal) or -1 (Anomaly).
The English Translation: We will catch any -1 scores and use Python to dynamically generate the exact sentence: "There is a sudden increase in the traffic from the below IP(s)..."

#2 Fake/Suspicious URL validation using AI analysis - (Here also mention which module is required in the lambda layer for this mechanism to work )
Because your legitimate API endpoints are long and contain a high variety of characters, numbers, and query parameters (like ?id=79890&limit=20&Offset=0), the Shannon Entropy algorithm calculated a high "randomness" score. When it paired that high score with the 0 2XX count, it generated a false positive, mistaking a complex API path for a scrambled DDoS string.
Here is exactly how we can improve the AI logic to make it highly accurate and eliminate these false positives:
1. Length-Normalized Entropy (The Math Fix)
Instead of just looking at the total randomness of the string, the algorithm needs to divide the entropy score by the length of the string.
A short string with high randomness (e.g., /qwxz9) is highly suspicious.
A long string with high randomness (e.g., /auth-service-binge/v2/oauth/...) is completely normal. Normalizing the score will fix the mathematical skew.
2. Isolate the URL Path (Strip the Noise)
Right now, the model is reading the entire line: https://tb.tapi.videoready.tv:443/homescreen-client/pub/api/.... The https, the domain, and the query parameters (?id=...) add massive amounts of baseline randomness. I will update the code to use Python's urllib.parse to strip away the domain and query parameters, forcing the AI to judge only the core structural path (e.g., /homescreen-client/pub/api/v4/rail/seeAll).
3. Lexical Keyword Heuristics (The Human Element)
We can inject a "whitelist" of common architectural terms directly into the AI's scoring matrix. If the URL path contains structural keywords like api, v1, v2, oauth, token, auth, login, or offset, the model will artificially drop its threat score. Scrambled DDoS hits like /ioyhcwn9iy[c0wuc0 do not contain clean English architectural words.
4. Consonant-to-Vowel Ratio Check
Scrambled, malicious strings (/shittttttt, /bitchhhhh, /dfghjkll) almost always fail basic English lexical structure. By calculating the ratio of vowels to consonants, the algorithm can easily distinguish between a human-readable API path and a random keyboard smash.
By combining the structural parsing, the keyword heuristics, and the normalized math, the AI will easily ignore your failing API endpoints and only trigger on genuine malicious garbage.
